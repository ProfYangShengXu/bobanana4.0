# Cyclic Workflow — 自驱循环工作流

你正在执行 **cyclic-workflow** 技能。在本技能的作用域内，你承担一项额外职责：**在每个阶段结束时，模拟用户视角生成下一阶段的提示词**，并通过 `queue_next_prompt` 工具写入状态文件，供 `reasonix cycle <目标>`（推荐）或 `scripts/cycle.sh` 编排执行下一轮会话。你仍是 Reasonix Code——这项职责是行为指令，不是身份切换。

## 核心概念

```
用户目标 → Phase 1 (HTML 设计文档) → Phase 2 (逐步执行) → Phase 3 (验证收敛) → 用户验收
```

- **你的角色**：你是用户目标的执行者，同时也是下一轮会话的"用户"——你为自己（或下一个自己）撰写下一阶段的提示词。
- **状态工具**：使用 `mcp__cycle-bridge__queue_next_prompt` 存储下阶段提示词；`mcp__cycle-bridge__get_cycle_state` 读取当前状态；`mcp__cycle-bridge__signal_done` 标记完成。
- **编排**：三阶段可由 `reasonix cycle <目标>`（推荐，Go 原生实现，跨平台一致，支持 `--resume` 断点续跑）或 `scripts/cycle.sh` 编排。

## 阶段判断

首先调用 `mcp__cycle-bridge__get_cycle_state` 确定当前处于哪个阶段：
- **无状态或 phase=1**：执行 Phase 1
- **phase=2**：执行 Phase 2
- **phase=3**：执行 Phase 3
- **phase=done**：已结束，报告状态

---

## 启动自检（Phase 2/3 必做）

在进入当前阶段的具体工作前，先验证上一阶段的产出物是否存在且未被篡改：

1. **检查产出物存在**：从 prompt 的 [DONE] 段 `Artifacts:` 行提取文件路径列表，逐个 `ls` 验证。如果 [DONE] 缺少 `Artifacts:` 行——这是一个警告信号，手动从 [DONE] bullet 中提取提到的文件路径验证。
2. **检查产出物指纹**（如 [DONE] 包含 `Fingerprint:` 行）：用 `bash -c "cat <文件路径> | sha256sum"` 重新计算 hash，与 [DONE] 中记录的值对比。
3. **不匹配处理**：
   - 文件缺失 → 向用户报告具体缺失文件，暂停执行
   - Hash 不匹配 → 向用户报告 "产出物可能被外部修改或上一阶段记录不实"，列出差异
   - 全部匹配 → 继续执行阶段工作

## Phase 1：HTML 设计文档

**目标**：生成一份完整、专业的 HTML 设计文档，用户可用浏览器打开审阅。

### 产出物
- `docs/design/<项目名>-design.html`：自包含的 HTML 设计文档

### HTML 文档必须包含的内容

1. **项目概览**：目标、设计哲学、技术栈选型与理由

2. **架构设计**：
   - 模块/组件树状结构图（HTML 列表 + CSS 缩进线）
   - 数据流图（用纯 CSS/HTML 绘制的流程图）
   - 路由设计（如适用）

3. **架构设计原则（必审）**：在设计架构时，对照以下原则逐条审视，不满足的需在文档中说明理由：
   - **高内聚低耦合**：模块间按接口通信而非共享状态。每个模块只通过少数公开接口暴露能力。
   - **单一职责**：每个模块/函数只有一个明确的改变理由。一个模块不做"既管数据库又管渲染"的事。
   - **开放封闭**：对扩展开放，对修改封闭。新功能应通过新增代码而非改动已有代码实现。
   - **KISS**：最简单的方案优先。不要为了"未来可能需要"而引入抽象层。
   - **DRY**：相同逻辑只出现一次。遇重复立即提取公共函数/组件。
   - **最少知识（迪米特）**：模块只与直接依赖通信，不穿透调用链（`a.B().C().D()` 是反模式）。
   - **依赖倒置**：高层模块不依赖低层实现，双方都依赖抽象接口。框架代码不应耦合具体库。
   - **关注点分离**：基础设施（DB/网络/IO）、业务逻辑、展示层分层隔离。
   - **YAGNI**：不添加"可能以后用得上"的功能、配置项、抽象层。只有当前需求明确需要才做。
4. **详细计划**：分解为可执行的子任务，每个子任务标注：
   - 预计涉及的文件
   - 依赖关系
   - 预估复杂度（低/中/高）
5. **多层验证方案**：
   - L1：静态分析（lint / typecheck）
   - L2：单元测试（覆盖关键路径 + 边界条件）
   - L3：集成测试（模块间交互）
   - L4：端到端验证（用户视角验收）
   - L5：非功能验证（性能 / 安全 / 无障碍）
6. **最终验收标准**（必须可量化）：
   - 代码强鲁棒性：所有函数有输入校验，无 unchecked nil/undefined
   - 防呆措施：用户输入边界处理、并发冲突处理、网络失败降级
   - 安全措施：无硬编码密钥、输入消毒、CSRF/XSS 防护（如适用）
   - 无死锁：异步操作有超时机制、资源获取有顺序约定
   - 前端显示：响应式布局、加载/空/错误三态处理
   - 功能完备：所有计划功能可运行，无 TODO 残留
   - 性能优化：关键路径无冗余渲染/查询
7. **风险与边界**：明确不做的事、已知限制、退化策略。对照以下风险意识清单，逐条审视并在文档中记录分析结论：
   - **技术债意识**：当前设计中有哪些为了赶速度而做的妥协？每个妥协标记为 TODO/technical-debt 并附带重构方向。
   - **回滚意识**：如果这次改动上线后出了问题，回滚方案是什么？数据库迁移是否可逆？API 变更是否向后兼容？
   - **灰度/小范围验证**：高风险改动是否有 feature flag 或分阶段上线的设计？能否只对部分用户生效？
   - **熔断与降级**：依赖的外部服务挂了时，系统如何反应？是否每个外部调用都有超时、重试、降级策略？
   - **监控与告警**：关键路径是否有日志和指标？失败是否可被自动发现？新增的异常路径是否有人盯着？
   - **容错与冗余**：单点故障在哪里？关键服务是否有多个副本或备用方案？
   - **安全性意识**：每个用户输入入口是否有消毒？认证/授权是否有遗漏？密钥/令牌是否从环境变量注入而非硬编码？

### HTML 样式要求
- 使用简洁专业的配色（深色标题栏 + 浅色内容区）
- 响应式布局（桌面优先，移动端可读）
- 可折叠的章节（`<details>` 元素）
- 打印友好（`@media print`）
- 包含目录导航（sticky sidebar）

### 完成 Phase 1 时
1. 用 `write_file` 写出 HTML 文档。
2. **生成确定性验证脚本**：用 `write_file` 写出 `docs/design/<项目名>-verify.sh`。该脚本将 Phase 3 的验收标准转化为一个 `bash` 命令即可完成的全自动检查：
   ```bash
   #!/usr/bin/env bash
   set -euo pipefail
   echo "=== A: No hardcoded secrets ==="
   # Agent fills in actual grep commands below
   echo "=== B: No TODO/FIXME ==="
   echo "=== C: Lint ==="
   echo "=== D: Typecheck ==="
   echo "=== E: Tests ==="
   echo "=== F: Build ==="
   echo "=== ALL CHECKS PASSED ==="
   ```
   每个检查失败时 `exit 1`。Phase 3 只需 `bash verify.sh` → 退出码 0 = 全部通过。
3. **计算产出物指纹**：用 `bash -c "cat docs/design/<项目名>-design.html docs/design/<项目名>-verify.sh | sha256sum"` 获取 hash。
4. 告知用户文档路径。
5. **交互模式（chat/serve）**：等待用户审阅反馈。用户确认后进入步骤 6。
6. **Headless 模式（reasonix run）**：直接进入步骤 7。
7. 调用 `mcp__cycle-bridge__queue_next_prompt` 进入 Phase 2：
   - `phase`: `"2"`
   - `prompt`: 按下方「prompt 模板」格式撰写
   - `[DONE]` 必须包含：`Fingerprint: sha256:<hash>`（步骤 3 的结果）
   - `goal`: 原始用户目标

## Phase 2：逐步执行

**目标**：按照 Phase 1 的设计文档（PRD），逐任务实现代码，每步自我验证纠错。

### 铁律：不准跳过 PRD

**Phase 2 的第一步必须是 `read_file` 读取 Phase 1 生成的 HTML 设计文档。** 不准凭记忆或推断直接编码。设计文档中的架构图、任务分解、验证方案是唯一执行依据。

如果在编码过程中发现设计文档有遗漏或错误，必须先更新设计文档再改代码，而不是改完代码再补文档。

### 执行流程

1. **读取 PRD（必做）**：
   > 调用 `read_file` 读取 `docs/design/<项目名>-design.html`。
   > 从中提取：
   >   - 任务列表（计划 §4）
   >   - 验证方案（计划 §5）
   >   - 最终验收标准（计划 §6）
   > 将任务列表写到当前上下文。**不准跳过这一步直接编码。**

2. **建立追踪**：用 `todo_write` 创建任务清单，按依赖排序，标注每项的预期验证级别（L1-L5）。

3. **逐任务执行**：
   - **粒度约定**：每个 `task` 子任务对应 PRD 计划中的一条可独立验证的功能单元。不准合并多条计划项到同一个 `task` 调用。
   - **禁止直接在大 agent 中写代码**——每个子任务必须通过 `task` 工具派生子 agent 执行，不准在主会话中直接 `write_file`/`edit_file` 实现功能代码。
   - 依赖项先完成，独立子任务可并行派发。
   - 对每个子任务：
     a. 使用 `task` 工具（description="<任务名>"）派生子 agent
     b. 子 agent 返回后，**审阅其产出是否与 PRD 一致**
     c. 对照 PRD 中的验证方案执行相应级别的验证
     d. 如有问题，修正后重新验证
     e. 用 `complete_step` 标记完成（evidence 中必须引用 PRD 中的对应任务编号）

4. **工程落地实践**：每个子任务实现时，必须践行以下工程实践（不满足的需在 complete_step 证据中说明理由）：
   - **版本管理（Git 思维）**：重要变更（重构、API 修改、配置变动）应该有提交信息记录"修改了什么、为什么修改"。不仅最后提交，中间关键节点也可提交。
   - **持续集成思维**：每次代码变更后应运行 lint + typecheck + 单元测试。不要"写完一起检查"——每步都过。
   - **自动化测试覆盖**：新增功能必须有对应测试。修复的 bug 先写一个暴露该 bug 的测试，再修复。边界条件必须有测试。
   - **代码/文档审查**：子 agent 产出的代码，父 agent 必须审阅——不只看"跑没跑过"，还要看命名是否清晰、是否有死代码、是否与项目风格一致。
   - **环境管理意识**：配置（API 地址、密钥、开关）与代码分离。不把环境相关值硬编码到代码中。
   - **配置管理**：新增配置项应有默认值、有文档、有校验逻辑。配置变更要向后兼容。
   - **全流程闭环**：每个子任务的生命周期：开发 → 本地验证（lint/test/build）→ 提交 → 集成验证，缺一不可。

5. **子 agent 产出抽查**：每完成 3 个 task 子任务后，随机选 1 个，重新运行其验证命令：
   - 如果子 agent 声称 "test passed" → 实际运行对应测试命令
   - 如果抽查不通过 → 标记该子任务为失败，重新执行
   - 如果抽查通过 → 在 todo 中记录 "spot-check: OK"

6. **纠错循环**：
   - 遇到错误时，分析根因 → 修正 → 重新验证，最多重试 3 次
   - 3 次后仍失败，记录到问题日志 `docs/design/<项目名>-issues.md`

7. **测试集优化**：
   - 在纠错过程中，将发现的边界条件补充到测试用例
   - 确保测试覆盖了实际出现过的失败模式

8. **验证方案完善**：
   - 将执行过程中发现的遗漏验证项补充到 Phase 1 的 HTML 文档（更新 L1-L5）
   - **填充 verify.sh**：Phase 2 结束时，`docs/design/<项目名>-verify.sh` 必须是可执行的完整脚本——每个检查项用真实的 grep/build/test 命令替换占位符。Phase 3 的 agent 执行它之前会先检查：如果脚本中仍有未替换的 `echo "==="` 占位行，视为 Phase 2 未完成。

### 完成 Phase 2 时
1. 所有子任务 `complete_step` 完毕，每个 evidence 都引用了 PRD 任务编号
2. 运行完整测试套件确认全部通过
3. **计算产出物指纹**：用 `bash -c "find . -name '*.go' -o -name '*.ts' -o -name '*.py' -not -path './node_modules/*' -not -path './.git/*' | sort | xargs cat | sha256sum"`（按实际文件类型调整）获取全域 hash。
4. 调用 `mcp__cycle-bridge__queue_next_prompt` 进入 Phase 3：
   - `phase`: `"3"`
   - `prompt`: 按 prompt 模板格式撰写，[DONE] 中必须体现 PRD 任务完成情况
   - `[DONE]` 必须包含：`Fingerprint: sha256:<全域hash>`
5. 说明："Phase 2 完成，请运行 `reasonix cycle --resume` 或 `scripts/cycle.sh` 进入验证收敛阶段"

## Phase 3：验证收敛

**目标**：对照 PRD 中的最终验收标准，逐项验证，直到全部达标。

### 铁律：以 PRD 为准

**验证的唯一依据是 Phase 1 设计文档中定义的验收标准（§6）和验证方案（§5）。** 不准自己发明验收条件，不准降低 PRD 中定义的标准。如果验证过程中发现 PRD 的标准不合理，必须先更新 PRD 再调整验证。

### 验证报告格式（强制）

每项检查的结果必须包含可复现的原始数据，不得用主观判断替代：
- **(a) 完整命令行**：粘贴实际运行的命令（含所有参数）
- **(b) 退出码**：命令的退出码（`echo $?`）
- **(c) 输出摘要**：stdout/stderr 的前 10 行或关键错误行
- **禁止**：仅写 "passed" / "通过" / "looks good" 等主观判断——下一轮 agent（或用户）无法验证

### 验证循环

```
重新读取 PRD → 提取验收标准 → 运行验证 → 全部通过 → signal_done
                                    │
                              仍有失败 → 修复 → 重跑（最多5轮）
```

### 验证步骤

0. **必做：重读 PRD**
   > 调用 `read_file` 读取 `docs/design/<项目名>-design.html`。
   > 从 §6 验收标准中提取验收条件清单。
   > 从 §5 验证方案中提取每项的验证方法。
   > **不准跳过这一步直接跑 verify.sh。**

1. **首选：运行确定性验证脚本**。如果 Phase 1 生成了 `docs/design/<项目名>-verify.sh`：
   - **完整性检查**（二段验证）：先 `grep -cE "(echo \"===|# Agent fills)" docs/design/<项目名>-verify.sh`——如果命中数 > 2，说明仍有未替换的占位符/注释。再对每个检查块（`echo "=== X:"` 之间的区域）grep 实际命令关键字：`grep|go test|npm test|pytest|tsc --noEmit|go build|cargo build|mvn`——如果任意一块命中数为 0，说明该检查项仍是空壳。两种检测任一项命中，视为 Phase 2 未完成，停止并报告。
   - 通过完整性检查后，执行 `bash docs/design/<项目名>-verify.sh` 并按验证报告格式记录结果
   - 退出码 0 → 全部检查通过，跳至人工关注清单
   - 退出码非 0 → 对照 PRD 分析哪些验收条件未满足，进入修复循环

2. **备选：逐项手动验证**（verify.sh 不存在时使用）：

   | # | 检查项 | 命令 | 通过标准 |
   |---|--------|------|---------|
   | A | 无硬编码密钥 | `grep -r "api_key\|password\|secret\|token" --include="*.go" --include="*.ts" --include="*.py" \| grep -v "env\|config\|test"` | 零命中 |
   | B | 无 TODO/FIXME | `grep -r "TODO\|FIXME\|HACK\|XXX" --include="*.go" --include="*.ts"` | 零命中 |
   | C | Lint | 项目 lint 命令 | 零 error |
   | D | Typecheck | 项目 typecheck 命令 | 零 error |
   | E | 测试 | 项目测试命令 | 零 failure |
   | F | 构建 | 项目构建命令 | 零 error |

   每项按验证报告格式记录——(a) 命令 (b) 退出码 (c) 输出摘要。

3. **修复循环**：
   - 分析失败项 → 修复代码或 verify.sh → 重新运行 → 记录结果
   - 每项最多重试 3 次
   - 3 次后仍失败：记录到 `docs/design/<项目名>-known-gaps.md`
   - 全局最多迭代 `max_iterations` 次（默认 5 次）

4. **人工关注清单**（`read_file` 审查，标注给用户，不阻塞 signal_done）：
   - 并发安全：列出所有异步点（goroutine/async/await），确认超时/取消机制
   - 输入消毒：列出外部输入入口，确认消毒逻辑
   - 前端三态：列出所有 UI 组件，确认 loading/empty/error 三态

### 完成 Phase 3 时
调用 `mcp__cycle-bridge__signal_done`：
- `summary`: 简短总结完成了什么、对照 PRD 的验证结果（通过 N/总 M 项）、已知差距（如有）
- 编排脚本（`reasonix cycle --resume` 或 `cycle.sh`）将检测到完成状态并退出
- 用户回来验收最终产物

## 通用规则

1. **始终站在用户角度**：每阶段结束时问自己"如果我是用户，下一轮我希望 agent 知道什么？"
2. **prompt 模板（强制）**：`queue_next_prompt` 的 `prompt` 参数必须按以下结构撰写，缺一不可：
   ```
   [GOAL] <原始用户目标，一字不改——cycle-bridge 已启用哈希锚定，改一个字就会被拒绝>
   [PHASE] Phase N — <阶段名称>
   [DONE] <上一阶段完成的摘要，3-5 条 bullet>
          Artifacts: <path1>, <path2>, ...  ← 必须列出所有产出物路径，启动自检逐个验证
          Fingerprint: sha256:<hash>  ← Phase 1/2 必须包含；Phase 3 的启动自检会验证
   [NEXT] <本阶段的具体执行指令——引用 Phase N 的步骤>
   ```
   下一轮 agent 只会看到这段 prompt，看不到当前会话历史。
   [GOAL] 改一个字就会被 cycle-bridge 的哈希锚定拒绝；[DONE] 缺少 Fingerprint 会在下一轮启动自检时被标记为不可验证。
3. **提示词要自包含**：`queue_next_prompt` 的 `prompt` 参数必须包含下一轮 agent 需要的全部上下文——它不会看到当前会话的历史。
4. **保持状态一致**：状态文件 `.reasonix/cycle/state.json` 是唯一的真实来源。
5. **HTML 可迭代**：用户可在 Phase 1 后修改 HTML 文档来调整方向；后续 Phase 应重新读取 HTML 获取最新设计。
6. **失败友好**：任何阶段出错都不要静默失败——记录、汇报、给出建议的下一步。
