# Iterative Loop — 小循环逼近

**轻量自驱循环：设一个目标，AI 不断迭代逼近，每次只改一件事，改完验证，不行再改。**

```
设定目标 → 评估现状 → 改一件事 → 验证 → 达标? → 完成
                   ↑                        │
                   └──── 未达标，继续 ────────┘
```

---

## 适用场景

- 已有代码需要**反复打磨**（覆盖率、性能、可读性）
- 一个指标不够好，需要逐步提升
- 知道"做成什么样"，但不确定"怎么做"——让 AI 自己试

## 不适用场景

- 从零开始的大项目 → 用 `/cycle` 三阶段
- 一次改完的简单任务 → 直接说

---

## 参数

| 参数 | 说明 | 默认 |
|------|------|------|
| target | **目标描述**——要达成什么状态、满足什么条件 | 必填 |
| scope | 作用范围——文件或目录 | 当前目录 |
| max_iterations | 最大迭代次数 | 5 |
| check_method | 达标判定：`auto`(AI判断) / `command`(退出码) / `manual`(用户确认) | auto |

---

## 铁律

### 铁律 1：不准漂移（target 锁定）

**进入循环后 target 就是铁律，不准自己改。**

- 如果迭代过程中发现 target 不合理或无法实现，必须在循环报告中记录"target 可能不合理"，**不能自己降低标准让循环提前结束**
- 每一轮评估差距时，必须**重新读取原始的 target 描述**，不准凭记忆用"优化过的版本"
- 差距清单中的每一项必须直接对应 target 的原始表述，不准自己发明新的评估维度

### 铁律 2：不准有空壳/死代码

**每次改进后，在 complete_step 前必须 grep 以下模式：**

| 模式 | 检查命令 | 处理 |
|------|---------|------|
| 空函数体 | `grep -rn "function .* {}\\|def .*: *pass\\|func .*{}"` | 打回重做 |
| 桩/存根 | `grep -rn "NotImplemented\\|todo!()\\|FIXME\\|TODO"` | 打回重做 |
| 空 return | `grep -rn "return nil$\\|return null$\\|return \"\"$"` | 确认或打回 |
| 未使用 import | 运行 lint（`eslint` / `go vet` / `pylint`） | 清理 |
| 虚假逻辑（只走 happy path） | 人工审视函数是否有 error 处理 | 对照 target 检查 |

### 铁律 3：不准幻觉验证

**验证结果必须基于实际命令的输出，不准自己替命令回答。**

```
check_method=auto:   展示 diff + 说明差距是否缩小
check_method=command: 必须运行命令 → 粘贴退出码 + 输出摘要
                     （"test passed" 不算，要有 "go test ./... → exit 0" 才算）
check_method=manual:  展示改动，等用户说"继续"或"停"
```

**禁止**以下验证形式：
- ❌ "看起来没问题" 
- ❌ "应该达标了"
- ❌ "代码质量有所提升"
- ❌ 声称跑了命令但没贴输出

**必须**的验证形式：
- ✅ `Command: go test -cover ./... → exit 0, coverage: 92%`
- ✅ `Command: grep "TODO" src/ → exit 1 (no matches)`

### 铁律 4：工程实践（每次迭代必过）

每轮迭代中，改动的代码必须满足：

| 原则 | 落地检查 |
|------|---------|
| **KISS** | 改动是否是最简单的方案？有没有引入不必要的抽象？ |
| **DRY** | 新代码是否重复了已有逻辑？如果是，是否提取了公共函数？ |
| **单一职责** | 改动的函数/组件是否只做一件事？ |
| **YAGNI** | 是否加了"以后可能用得上"的额外功能？删掉。 |
| **命名清晰** | 变量/函数名是否准确描述其作用？ |

## 循环逻辑

```
循环开始
  │
  ├─ 步骤 1: 🧐 评估现状
  │   read_file 读取目标文件
  │   对照原始 target 输出差距清单（bullet list，标注未达标 / 部分达标 / 已达标）
  │   差距清单中每项都要引用 target 的原始表述，不准自己发明
  │
  ├─ 步骤 2: 🎯 选一项改进
  │   从差距清单中选择 **改动最小、收益最大** 的一项
  │   说明：改哪个文件、改什么、为什么、预期效果
  │   **每次只改一件事。** 不准合并多项改动。
  │
  ├─ 步骤 3: 🔧 执行
  │   write_file / edit_file 做具体改动
  │   **改完后立即清理**：删未用 import、死变量、残留 TODO
  │
  ├─ 步骤 4: ✅ 验证
  │   按铁律 3 的格式输出验证结果
  │   按铁律 2 grep 检查死代码
  │   按铁律 4 检查工程实践
  │
  ├─ 差距缩小? → 继续循环
  │   todo_write 记录：「迭代 N: 改了X → 差距从A缩到B」
  │   连续 2 次无进展 → 报告"陷入局部最优，建议换方案"
  │
  └─ 达标或超限 → 结束
      展示：改动清单、差距变化轨迹、最终结果
```

## 完整示例

```
target: server.go 的测试覆盖率从 40% 提升到 90%
check_method: command → go test -cover ./...

迭代 1: 添加空 name 测试用例
  Command: go test -cover ./... → exit 0, coverage: 55%
  grep 死代码: 零命中 ✅
  差距: 从 40%→55%，还有 35% 差距

迭代 2: 添加长名称边界测试
  Command: go test -cover ./... → exit 0, coverage: 70%
  grep 死代码: 零命中 ✅
  差距: 55%→70%

...

迭代 5: 所有分支覆盖
  Command: go test -cover ./... → exit 0, coverage: 93%
  ✅ 达标，结束
```

## 用法

```
/skill iterative-loop target="server.go 覆盖到 90%" max_iterations=5
```
