# Iterative Loop — 小循环逼近

**一个轻量级的自驱循环：设一个目标，AI 不断迭代逼近它，每次都比上次好一点。**

```
设定目标 → 评估现状 → 改进 → 验证 → 达标? → 完成
                   ↑                     │
                   └── 未达标，继续 ───────┘
```

## 适用场景

- 已有代码/文本/设计，需要**反复打磨**直到满意
- 一个指标不够好（性能、可读性、覆盖率），需要逐步提升
- 不确定具体怎么做，但知道"做成什么样"——让 AI 自己试

## 不适用场景

- 从零开始的大项目（用 `/cycle` + cyclic-workflow 三阶段更合适）
- 只需要一次修改的简单任务（直接说就行了，不用进循环）

---

## 参数

| 参数 | 说明 | 默认 |
|------|------|------|
| target | 目标描述——要达成什么状态、满足什么条件 | **必填** |
| scope | 作用范围——文件、目录或描述 | 当前目录 |
| max_iterations | 最大迭代次数 | 5 |
| check_method | 达标判定方式：auto（AI 判断）/ command（退出码）/ manual（用户确认） | auto |

---

## 循环逻辑

```
循环开始
  │
  ├─ 步骤 1: 🧐 评估现状
  │   读取目标文件/数据，对照 target 评估当前差距
  │   输出差距清单（bullet list，每项标注未达标 / 部分达标 / 已达标）
  │
  ├─ 步骤 2: 🎯 行动计划
  │   基于差距清单，选择 **改动最小、收益最大** 的一项改进
  │   说明：改什么、为什么、预期效果
  │   **每次只改一件事。** 不准一次改多个点——否则无法判断哪项改进有效。
  │
  ├─ 步骤 3: 🔧 执行改进
  │   用 task 子 agent 或直接 write_file/edit_file 执行具体改动
  │   （小循环不像三阶段那样禁 write_file——可以自己改，但要小而准）
  │
  ├─ 步骤 4: ✅ 验证结果
  │   重新评估目标文件，对照 target 检查：
  │   - check_method=auto: 你自己判断是否达标
  │   - check_method=command: 运行指定命令，退出码 0 = 达标
  │   - check_method=manual: 展示改动的 diff，等用户确认
  │
  ├─ 差距仍存在? → 循环迭代
  │   todo_write 记录进度：当前迭代次数 / 总次数
  │   如果连续 2 次迭代没有缩小差距 → 停止并报告"陷入局部最优"
  │
  └─ 达标或超限 → 结束
      展示最终结果：改了什么、差距清单、迭代次数
```

## 铁律

1. **每次只改一件事。** 不准在一次迭代中改多个文件/多个函数。小循环的核心是"走一步看一步"。
2. **不准悄悄放松 target。** 如果发现 target 不合理，先报告用户再调整——不准自己降低标准让循环提前结束。
3. **连续 2 次无进展就停。** 如果迭代 2 次后差距清单没有缩短，说明当前方法无效，停止循环并报告"陷入局部最优，可能需要换思路"。
4. **不改没问题的代码。** 差距清单中已达标的不准回退。

## 示例

```
target: 将 greet() 函数的测试覆盖率从 40% 提升到 90%
max_iterations: 5
check_method: command → go test -cover

迭代 1: 添加了 name 为空时的测试 → 覆盖率 55%
迭代 2: 添加了长名称边界测试 → 覆盖率 70%
迭代 3: 添加了特殊字符测试 → 覆盖率 85%
迭代 4: 添加了并发调用测试 → 覆盖率 92% ✅ 达标
```

/skill iterative-loop target="xxx" max_iterations=5
